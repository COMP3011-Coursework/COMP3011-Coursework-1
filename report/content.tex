\section{Introduction}

Global food insecurity remains a persistent humanitarian concern; commodity price
volatility is an early indicator of supply disruptions and crises.
This report describes the \textbf{Global Food Price Monitor API}, a full-stack
web service built on the WFP Global Food Prices dataset \parencite{hdx_wfp} that
provides RESTful CRUD endpoints, analytical aggregation endpoints, an interactive
choropleth frontend, and an MCP server for AI-assistant integration
\parencite{mcp_spec}.

The dataset \parencite{hdx_wfp} covers 98~countries, \textasciitilde3,000~markets,
and \textasciitilde3.14~million price observations from 1992 to the present.
It is licensed under CC~BY-IGO, permitting unrestricted academic use with attribution.

\section{Design}

\subsection{System Architecture}

The application follows a three-tier architecture: a React SPA frontend, a
Python/FastAPI backend, and a PostgreSQL database.
The backend also mounts an MCP endpoint at \texttt{/mcp} (Figure~\ref{fig:architecture}).

\subsection{Technology Stack}

Technology choices were guided by structured GenAI exploration of alternatives
alongside industry best practice.
\textbf{PostgreSQL} was chosen for its window functions and CTE support, essential
for the crisis-score computation.
\textbf{SQLAlchemy} \parencite{sqlalchemy_docs} provides an async ORM; \textbf{Alembic}
\parencite{alembic_docs} handles versioned schema migrations.
\textbf{FastAPI} \parencite{fastapi_docs} was preferred over Django REST Framework
for its lower overhead and automatic OpenAPI documentation generation from type
annotations; \textbf{Pydantic} \parencite{pydantic_docs} enforces strict runtime
type coercion at the API boundary.
\textbf{React/TypeScript} \parencite{react_docs} delivers a component-driven SPA
with Leaflet.js for mapping and Chart.js for time-series charts.
\textbf{Docker Compose} \parencite{docker_docs} orchestrates all services locally.

\subsection{Database Design}

Five tables comprise the schema: \texttt{prices} (fact), \texttt{markets},
\texttt{commodities}, \texttt{currencies}, and \texttt{users}.
\texttt{prices} stores both the local-currency price and a pre-computed USD price,
enabling cross-country comparability without repeated currency conversion at query time.
Three composite indexes on \texttt{(countryiso3, date)},
\texttt{(commodity\_id, date)}, and \texttt{market\_id} target the most
common analytical access patterns; these were identified empirically after profiling
the analytics queries on the full dataset.
On first startup, an idempotent seeding routine downloads the WFP CSV files and
bulk-inserts all records; subsequent restarts skip seeding if data is already present.

\subsection{API Design}

The REST API operates under \texttt{/api/v1} across
four routers:
\textbf{Auth} (\texttt{/auth}) — JWT Bearer tokens \parencite{rfc7519};
\textbf{Prices} (\texttt{/prices}) — full CRUD (authenticated writes);
\textbf{Analytics} (\texttt{/analytics}) — five read-only analytical endpoints;
\textbf{Reference} (\texttt{/countries}, \texttt{/commodities},
\texttt{/markets}, \texttt{/currencies}) — lookup tables.
Interactive documentation is auto-generated at \texttt{/docs} (Swagger~UI) and
\texttt{/redoc} (ReDoc); a PDF export is at \texttt{docs/api-reference.pdf}.

Error handling follows HTTP conventions throughout: \texttt{400} for malformed
requests, \texttt{401} for missing or invalid tokens, \texttt{404} for missing
resources, \texttt{422} for failed foreign-key validation, and \texttt{500} for
unexpected server errors.
All responses use consistent JSON payloads with a \texttt{detail} field describing
the error, following FastAPI's default validation error format.
CORS is configured to restrict origins to trusted domains in production,
preventing cross-origin misuse of authenticated endpoints.

\section{Features}

\subsection{Core CRUD Operations}

Authenticated users may create, read, update, and delete price observations via
standard HTTP verbs.
Write operations validate all foreign-key references, returning \texttt{422} on
constraint violations and \texttt{404} for missing records.
The list endpoint supports pagination (up to 200 records per page) and filtering
by country, commodity, market, and date range.

\subsection{Analytics Endpoints}

Five analytical endpoints augment the CRUD interface:
(1)~\textbf{Price Trends} — monthly average USD prices;
(2)~\textbf{Volatility} — commodities ranked by coefficient of variation
    ($\mathrm{CV} = \sigma/\mu$);
(3)~\textbf{Regional Comparison} — cross-country average USD prices for a commodity;
(4)~\textbf{Crisis Score} — composite food-crisis indicator (0--100);
(5)~\textbf{Market Summary} — latest prices per commodity at a market.

\paragraph{Crisis Score.}
Three components are computed over a trailing 12-month window:
$v$ (volatility) = mean CV across commodities with $\geq3$ observations;
$t$ (trend) = 3-month mean / 12-month mean $-1$;
$b$ (breadth) = fraction of commodities with a rising price.
Each is min-max normalised across countries, then combined as:
\begin{equation}
    s = (0.40\,v + 0.35\,t + 0.25\,b)\times 100.
    \label{eq:crisis}
\end{equation}
Severity bands: \textit{stable} ($s<25$), \textit{moderate} ($25\leq s<50$),
\textit{high} ($50\leq s<75$), \textit{critical} ($s\geq 75$).

\subsection{MCP Server and Frontend}

An MCP server at \texttt{/mcp} (FastMCP \parencite{fastmcp_docs}) exposes five
tools mirroring the analytics endpoints, enabling AI assistants to query the
system in natural-language workflows.
The frontend provides a choropleth world map colour-coded by crisis score, a
data explorer, and a data-entry form.
GeoJSON boundary data (\textasciitilde14\,MB) is cached client-side after first load.

\section{Implementation, Testing, and Deployment}

\paragraph{Development.}
An iterative GenAI-assisted workflow was used throughout.
A design document was drafted manually then refined through Claude dialogue
covering schema, endpoint design, authentication, and the crisis-score formulation.
Claude Code (Anthropic's CLI agentic tool) implemented components against the
specification; all output was reviewed and tested by the developer.
Authentication uses JWT Bearer tokens \parencite{rfc7519} with bcrypt password hashing.

\paragraph{Testing.}
The pytest suite \parencite{pytest_docs} uses FastAPI's \texttt{TestClient} backed
by \texttt{httpx} to exercise all HTTP endpoints end-to-end against a dedicated
in-memory SQLite test database, achieving \textbf{100\% line coverage} across all
backend modules (\texttt{pytest-cov}).
The test suite is structured into class-per-resource files:
\texttt{test\_prices.py} covers all CRUD paths and list-filter combinations;
\texttt{test\_analytics.py} covers all five analytical endpoints;
\texttt{test\_auth.py} covers registration, login, token validation, expiry, and
invalid-credential cases;
\texttt{test\_reference.py} covers lookup endpoints;
\texttt{test\_mcp.py} covers all five MCP tool wrappers end-to-end.
Shared fixtures (database session, test user, auth headers, commodity/market/currency
records) are defined in \texttt{conftest.py} and reused across test classes to
keep tests concise and avoid duplication.
The frontend was tested manually across browsers; MCP integration was verified
by connecting Claude Code and Claude Desktop to the live \texttt{/mcp} endpoint
and issuing representative natural-language queries.

\paragraph{Deployment and Performance.}
Docker Compose orchestrates all services locally; cloud deployment uses Render.com
with a managed PostgreSQL instance.
Locally (3.14M records): list pages \textasciitilde50\,ms; world map under~1\,s.
Render free tier (540k records): pages under 200\,ms; crisis scores \textasciitilde5\,s.
The live frontend\footnote{\url{https://comp3011-coursework-1-frontend.onrender.com}}
and API documentation\footnote{\url{https://comp3011-coursework-1-backend.onrender.com/docs}}
are publicly accessible.

Version control follows the Conventional Commits specification
\parencite{conventional_commits} with over 60 commits.

GitHub repository was released before the oral presentation.
Feature branching was not used: as a solo project with no concurrent development
streams, a linear trunk-based history provided clarity without sacrificing traceability.

\subsection{Challenges and Lessons Learned}

Several technical challenges arose during development.

\paragraph{Async SQLAlchemy.}
Adopting fully asynchronous database access (\texttt{AsyncSession}) required
careful attention to session lifecycle, particularly for the auto-seeding routine
that runs during application startup outside the normal request/response cycle.
A dedicated \texttt{AsyncSessionLocal} factory was introduced for background
tasks, separate from the request-scoped dependency, to avoid session leaks.

\paragraph{GeoJSON rendering.}
The choropleth map requires a 14~MB GeoJSON boundary file.
Fetching it on every page load made the map unusable; the solution was
client-side caching in React state so the file is downloaded once per session.
This highlighted the importance of distinguishing static assets from dynamic
API data in frontend architecture.

\paragraph{Emerging protocol integration.}
The MCP specification was relatively new at the time of development, with limited
documentation and few worked examples.
Understanding the protocol required sustained dialogue with Claude to explore
tool registration, session management, and transport options before selecting
FastMCP as the implementation library.
This was a deliberate exercise in using GenAI to learn an unfamiliar technology
quickly rather than waiting for mature documentation.

\paragraph{AI agent workflow.}
Using Claude Code as an agentic coding tool introduced workflow challenges absent
from conventional development.
The agent operates across multiple files simultaneously and may introduce changes
in unexpected locations; reviewing every diff carefully was essential to maintaining
a coherent codebase.
Managing context window limits across long sessions required breaking tasks into
focused sub-sessions and maintaining a design document as a shared reference.
This experience highlighted that effective AI-assisted development demands stronger
up-front specification discipline than solo manual coding.

\section{Limitations and Future Work}

The Render.com free tier restricts the cloud dataset to pre-2016 records.
The crisis-score algorithm (Equation~\eqref{eq:crisis}) is a custom heuristic not
validated against established indices such as the IPC scale; future work should
benchmark it against ground-truth assessments.
The authentication model is binary; role-based access control would be needed in
production.
The crisis-score computation is not server-side cached; a short-TTL cache would
reduce database load.
The frontend does not animate score evolution over time, which would enhance
interpretability.

\section{Conclusions}

The Global Food Price Monitor API meets and substantially exceeds the minimum
technical requirements.
The system delivers full CRUD on a 3-million-record humanitarian dataset, five
analytical endpoints including a novel composite crisis-score model, MCP-compatible
AI integration, 100\% automated test coverage, auto-generated API documentation,
an interactive choropleth frontend, and live cloud deployment.
The project demonstrates that directed use of GenAI tools---specifically Claude
Code for agentic development---can substantially accelerate delivery of
architecturally sound software, provided the developer retains genuine ownership
of design decisions and system understanding.
